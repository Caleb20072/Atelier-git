Concept fondamental

Git, c'est quoi exactement ?

Git est un système de contrôle de version distribué. Il prend des photos de votre code à chaque étape, vous laisse naviguer dans le temps, et permet à plusieurs personnes de travailler ensemble sans se marcher dessus.

Les 3 zones à connaître

1

Working Directory — votre espace de travail

Vos fichiers tels que vous les voyez dans votre éditeur. Vous modifiez, créez, supprimez. Git observe mais ne fait rien encore.

2

Staging Area (Index) — la salle d'attente

Vous choisissez quels changements feront partie de la prochaine photo. Commande : git add

3

Repository — l'album photo permanent

La photo est prise, estampillée avec votre nom + date + message. Immuable. Commande : git commit

git commit -m "feat: ajoute la page d'accueil et le script JS"

02
Branches

Travailler en parallèle
sans se faire peur

Une branche = une ligne de travail indépendante.
Expérimentez sans risquer votre code stable.

Analogie : Imaginez un livre dont vous écrivez plusieurs versions en parallèle. La branche main est l'édition officielle. Vous créez une branche pour tester un nouveau chapitre — si ça marche, vous l'intégrez. Sinon, vous l'abandonnez, sans avoir jamais touché l'édition officielle.

01

03
Merge

Fusionner les branches
sans perdre la tête

Le merge, c'est rassembler le travail de plusieurs branches.
Parfois fluide. Parfois conflictuel. Toujours maîtrisable.

Fast-forward merge

La branche principale n'a pas avancé. Git "avance la tête" proprement, sans commit de merge.

3-way merge

Les deux branches ont avancé indépendamment. Git crée un commit de merge qui réconcilie tout.

Merge conflict

Deux modifications incompatibles sur le même endroit. Git vous demande de trancher.

Fast-Forward Merge

Le merge le plus simple

Quand main n'a pas bougé pendant que vous travailliez sur votre branche, Git n'a qu'à "avancer le pointeur".

~/projet — sur main
git merge feature/ma-feature
Updating 3a1b2c..d4e5f6
Fast-forward
index.html | 12 +++++++
1 file changed, 12 insertions(+)
3-way Merge

Quand les deux branches ont avancé

Git compare 3 versions : l'ancêtre commun, la branche A, la branche B. S'il n'y a pas de ligne contradictoire, il fusionne automatiquement.

Ancêtre commun

function saluer() {
return "Bonjour";
}
→

Branche main

// Ajout commentaire
function saluer() {
return "Bonjour";
}
+

Branche feature

function saluer() {
return "Hello World";
}
↓

Résultat fusionné

// Ajout commentaire
function saluer() {
return "Hello World";
}

Merge Conflict

Quand Git ne peut pas décider seul

Un conflit arrive quand deux branches modifient le même endroit différemment. Git marque les zones et vous demande de choisir.

main dit :

function saluer() {
return "Salut !";
}
VS

feature dit :

function saluer() {
return "Coucou !";
}
CONFLICT
function saluer() { <<<<<<< HEAD return "Salut !"; ========= return "Coucou !"; >>>>>>> feature/ma-feature }

<<<<<<< HEAD — début de votre version (branche cible)

========= — séparateur entre les deux versions

>>>>>>> feature/... — fin de la version entrante

Votre rôle : choisir ou combiner, puis supprimer les marqueurs

Résolution de conflit — Interactif

À vous de jouer

Voici un fichier en conflit. Résolvez-le en écrivant le résultat final dans la zone de texte.

// config.js const CONFIG = { <<<<<<< HEAD (main) port: 3000, host: "localhost", ========= port: 8080, host: "0.0.0.0", >>>>>>> feature/deploy-config debug: true };

Contexte : main = config dev locale, feature/deploy-config = config de déploiement. Décision de l'équipe : garder le port 8080 et l'host 0.0.0.0.

Écrivez le fichier résolu :

Après résolution manuelle
Après avoir édité et sauvé le fichier sans les marqueurs
git add config.js
git commit -m "fix: résolution conflit config port/host"
[main a1b2c3] fix: résolution conflit config port/host

04
Workflows

Organiser le travail
en équipe

Comment structurer les branches pour travailler à plusieurs sans se rentrer dedans. Des conventions qui changent tout.

Pourquoi des conventions ? Sans règles communes, chaque dev a sa propre façon de nommer les branches, de merger, de déployer. Le résultat : personne ne comprend l'historique de personne. Les workflows résolvent ça.
GitHub Flow

GitHub Flow : idéal pour commencer

Un workflow minimaliste en 6 étapes, parfait pour les équipes qui déploient souvent.

1

main est toujours déployable

Règle absolue. Personne ne push directement sur main. Ce qui est sur main = ce qui est en production.

2

Créez une branche pour chaque feature/fix

Nom descriptif : feature/authentification, fix/bug-login, chore/update-deps

3

Committez régulièrement avec des messages clairs

Convention : feat:, fix:, docs:, chore:, test: (Conventional Commits)

4

Ouvrez une Pull Request (PR)

Demandez une revue de code avant de merger. Ça force à vérifier son propre travail.

5

Discutez et ajustez

Le reviewer commente, vous corrigez sur la même branche. Les nouveaux commits s'ajoutent à la PR.

6

Merge dans main + déploiement

Une fois approuvée, la PR est mergée. On supprime la branche feature.

Collaboration

Travailler avec un remote

Un remote (GitHub, GitLab…) est la copie partagée de votre repo.

git clone

Copie un repo distant sur votre machine. Inclut tout l'historique.

git pull

Récupère et intègre les derniers changements du remote.

git push

Envoie vos commits locaux vers le remote.

git fetch

Récupère les changements du remote sans les intégrer. Pour voir avant de merger.

git remote -v

Affiche les remotes configurés (URL de push/fetch).

git stash

Met de côté vos modifications non-committées pour changer de branche rapidement.

Récupérer le travail d'un collègue
git pull origin main
From github.com:equipe/projet
* branch main -> FETCH_HEAD

Pousser votre branche feature
git push -u origin feature/ma-feature
Branch 'feature/ma-feature' set up to track 'origin/feature/ma-feature'
.gitignore

Ce que Git ne doit jamais voir

Le fichier .gitignore liste les fichiers/dossiers à ne jamais commiter.

Secrets et clés

.env, secrets.json, clés API, mots de passe, certificats privés

Dépendances

node_modules/, vendor/, venv/ — régénérables, inutilement lourds

Fichiers générés

dist/, build/, *.o, *.class — produits de compilation

Fichiers d'IDE

.vscode/, .idea/, .DS_Store — spécifiques à votre machine

Exercice 3 — 20 min — Binôme

Provoquer et résoudre un vrai conflit

Exercice en binôme : Personne A crée le repo sur GitHub et invite Personne B comme collaborateur. Les deux clonent le même repo.

Personne A fait :

  1. Crée info.txt, ajoute "Version: 1.0.0", commit sur main
  2. Push sur GitHub
  3. Crée branche feature/version-alice
  4. Modifie : "Version: 2.0.0-alpha"
  5. Commit + push

Personne B fait :

  1. Clone le repo de A
  2. Crée branche feature/version-bob
  3. Modifie : "Version: 1.5.0-beta"
  4. Commit + push
1

A merge sa branche dans main — aucun problème

B fait git pull origin main pour se mettre à jour.

2

B tente de merger sa branche — conflit !

La ligne "Version" a été modifiée différemment. Git ne peut pas choisir seul.

3

A et B discutent et choisissent ensemble la version finale

B résout le conflit manuellement, commit, push.

Les commandes d'urgence

Tout le monde fait des erreurs. Ces commandes vous sauvent.

SituationCommandeAttention
"J'ai commité un mauvais message"git commit --amend -m "nouveau message"Seulement si pas encore pushé
"J'ai ajouté un fichier par erreur"git restore --staged fichier.txtRetire du staging, garde les modifs
"Je veux annuler mes modifications"git restore fichier.txtIrréversible
"J'ai commité sur la mauvaise branche"git reset HEAD~1 puis changer de brancheSeulement si pas encore pushé

"Je veux voir ce qui a changé"git diff / git diff --stagedAvant/après le add
